## 1. 定義

**AI-Driven GDR ビルドアップ**（以下「GDR ビルドアップ」）は、AI との対話を通じて GDR（General Decision Record）を反復的に積み上げながらプロジェクトを前進させる開発プロセスである。

従来の「設計 → 実装 → ドキュメント」のウォーターフォール型でも、TDD のように「先にテスト → 実装」の駆動型でもない。**対話の中で判断が結晶化し、GDR として定着し、実装に落ちる** — この積層サイクルそのものが方法論である。

## 2. 従来手法との比較

| 手法 | 起点 | 判断の記録タイミング | AI との相性 |
|---|---|---|---|
| ウォーターフォール | 設計文書 | 設計フェーズで事前作成 | 低（長大な文書を先に書く必要がある） |
| TDD | テスト | テスト作成時に暗黙的 | 中（テスト生成は可能だが判断根拠が残らない） |
| ADR 併用アジャイル | ストーリー | 実装後に追認で記録 | 中（記録忘れが起こりやすい） |
| **GDR ビルドアップ** | **対話** | **対話中にリアルタイムで結晶化** | **高（AI が判断を構造化し、その場で GDR 化する）** |

### 2.1. なぜ「GDR 駆動開発」と呼ばないのか

TDD（テスト駆動開発）の命名慣習に倣えば「GDR-Driven Development」が自然に思える。しかし「駆動」は **GDR を先に書き、それを満たすように実装する** というサイクルを暗示する。実態はそうではない。GDR は対話の中で事後的に結晶化するものであり、事前に書いてから開発を始めるものではない。最初の提案が過剰設計になり、人間の一言で圧縮されることもある。実装中にバグが見つかり GDR が書き換わることもある。この「対話と実装を通じて判断が積み上がっていく」プロセスを正確に表すのは「駆動（Driven）」ではなく「ビルドアップ（Build-Up）」である。

### 2.2. AI-Driven の位置づけ

開発を駆動するのは GDR ではなく **AI** である。この全体像を一語で表すのが **AI-Driven GDR ビルドアップ** である。AI が開発を駆動し（コード生成・設計提案・実装・検証）、その過程で生まれる判断が GDR として積み上がる。両者は不可分であり、AI 駆動から GDR を外すと判断が揮発し、GDR から AI 駆動を外すと単なる文書作業になる。

## 3. 期待される効果

### 3.1. 基本効果

1. **判断の揮発を防ぐ** — 口頭やチャットで決まったことが記録されずに消える問題がなくなる。対話がそのまま GDR として永続化される
2. **属人性の排除** — 「なぜこうなっているか」が GDR に残るため、担当者が変わっても判断を追跡できる。AI も過去の GDR を参照して一貫した提案を行える
3. **意思決定の速度向上** — AI が代替案・トレードオフ・影響範囲を構造化するため、人間は「選ぶ」ことに集中できる。判断に必要な情報の収集と整理を AI が担う
4. **過剰設計の早期検出** — AI の提案に対して人間が「最適解を」と一言返すだけで、ROI の低い施策が刈り込まれる。判断の根拠が定量化されているため、過剰かどうかの議論が客観的になる
5. **再検討の仕組み化** — 各 GDR に「再検討条件」が明記されるため、状況が変わった時に「見直すべき判断」が自動的に特定される。再検討のトリガが人間の記憶ではなく文書に依存する
6. **オンボーディングの加速** — 新規参加者がプロジェクトの判断履歴を GDR_INDEX から辿れる。「なぜこの設計なのか」を聞く代わりに GDR を読めばよい

### 3.2. AI の既知の失敗モードに対する抑止効果

AI にはいくつかの名前の付いた失敗モードがある。GDR ビルドアップの各メカニズムがそれぞれに対して抑止効果を持つ:

| 失敗モード | 説明 | GDR ビルドアップによる抑止 | 効果 |
|---|---|---|---|
| **ハルシネーション** | 事実でない情報をもっともらしく生成する | GDR の「定量化」原則により、実コードの計測データを根拠に組み込む。根拠のない主張は GDR フォーマットの「理由」欄で露呈する | ◎ |
| **コンテキスト喪失** | 長い対話で初期の文脈を忘れる | GDR が判断を永続化するため、対話が途切れても過去の決定を参照できる。セッションをまたいでも一貫性が保たれる | ◎ |
| **過剰同意（Sycophancy）** | ユーザーの意見に過度に同調し、問題を指摘しない | GDR の「代替案の提示」と「再検討条件」が構造的に反論・留保を要求する。「なぜやらないか」の記述が同調圧力を緩和する | ○ |
| **過剰設計（Over-Engineering）** | 不必要に複雑な解決策を提案する | 人間の「最適解を」の一言で ROI ベースの圧縮がトリガされる。GDR の定量化が過剰さを客観的に可視化する | ◎ |
| **矛盾生成** | 過去の出力と矛盾する新しい出力を生成する | 過去の GDR が永続化されており、新しい提案との整合性チェックが可能。AI 自身も過去の GDR を参照して一貫性を保てる | ◎ |
| **スコープクリープ** | 要求されていない機能や変更を勝手に追加する | GDR の「影響」欄が変更範囲を明示し、レビューで逸脱を検出できる。Phase 分けがスコープを区切る | ○ |
| **自信過剰（Overconfidence）** | 不確実な判断を断定的に述べる | GDR の「再検討条件」が「この判断がいつ崩れうるか」を明示的に問う。断定を留保に変換する構造的な力がある | ○ |
| **反復ループ** | 同じ誤りや提案を繰り返す | GDR_INDEX に過去の判断が蓄積されているため、同じ議論の再発を検出できる。「既に GDR-XXX-NNN で決定済み」と参照できる | ◎ |

**凡例:** ◎ 構造的に強い抑止効果 / ○ レビューサイクルと併用で効果を発揮

### 3.3. デメリット・留意点

GDR ビルドアップは万能ではない。導入時に認識しておくべきコストとトレードオフを示す:

| デメリット | 説明 | 緩和策 |
|---|---|---|
| **トークン消費の増加** | GDR の構造化・代替案の列挙・過去 GDR の参照により、AI とのやり取りで消費するトークン量が増える。特にコンテキストに過去の GDR を多数含める場合に顕著 | 参照する GDR を必要最小限に絞る。要約やインデックスを活用し、全文読み込みを避ける |
| **文書管理の負荷** | GDR が蓄積するほど、ステータス管理・相互リンク・採番の整合性維持に手間がかかる | AI に整合性チェックを委ねる（`整合性チェック` プロンプト）。定期的に Superseded な GDR をアーカイブする |
| **過剰な形式化** | 小さな判断にも GDR を書こうとすると、形式が目的化して開発速度が落ちる | GDR 化する判断の粒度をチームで合意する。明らかに自明な判断は GDR にしない |
| **AI 依存のリスク** | プロセスが AI 前提で設計されているため、AI が使えない状況（障害・コスト・規制）で回しにくい | GDR の書式と原則自体は人間だけでも運用できる。AI はあくまで効率化手段であり、必須ではない |
| **学習コスト** | scope / PREFIX / テンプレート / 短縮キーワードなど、チーム全体が理解すべき概念が多い | 最小構成（書式 + サイクル）から始め、ガイドやテンプレートは必要に応じて段階的に導入する |

## 4. サイクル

```
     ┌─────────────────────────────────────────┐
     │                                         │
     ▼                                         │
  1. 要件・課題を伝える                          │
     │                                         │
     ▼                                         │
  2. AI が分析し、判断を GDR として構造化          │
     ├─ scope / 代替案 / トレードオフを明示       │
     └─ 実装計画（Phase）を提案                  │
     │                                         │
     ▼                                         │
  3. レビュー・フィードバック                      │
     ├─ 承認 → 次へ                             │
     └─ 修正 → GDR を更新して再提案              │
     │                                         │
     ▼                                         │
  4. 実装                                       │
     ├─ AI がコードを書く                        │
     ├─ テスト・ビルドで検証                      │
     └─ GDR のステータスを「実装済み」に更新       │
     │                                         │
     ▼                                         │
  5. 次の要件・課題 ─────────────────────────────┘
```

各サイクルで GDR が 1 つ以上積み上がる。プロジェクトの判断履歴は GDR_INDEX に蓄積され、将来の意思決定の参照点となる。

## 5. 3 つの原則

### 5.1. 対話が設計書を生む（Dialog-First）

設計書を書いてから対話するのではなく、**対話の中で設計が生まれる**。AI は自然言語の要件を受け取り、判断の構造化（決定 / 理由 / 代替案 / 影響 / 再検討条件）を担う。人間は判断の方向性と受容可否に集中する。

### 5.2. 判断は積み上がる（Incremental Crystallization）

1 回の対話で完璧な GDR が生まれる必要はない。最初は粗い判断でスタートし、レビュー・フィードバック・実装を経て精度が上がっていく。

- 改善提案 → レビュー → レビュー反映 → 再レビュー → 実装
- 最初の GDR が後から別の GDR に分割されることもある
- 実装中に発見された制約が GDR の再検討条件を満たし、判断が更新されることもある

### 5.3. 実装が判断を検証する（Implementation Validates Decision）

GDR は仮説である。実装とテストによって初めて検証される。検証を経た GDR のステータスは「実装済み」に変わり、信頼性が上がる。検証で問題が見つかれば GDR を更新する。

## 6. AI に求められる振る舞い

| 役割 | 具体的な行動 |
|---|---|
| **構造化** | 自然言語の要件を GDR フォーマット（決定 / 理由 / 代替案 / 影響 / 再検討条件）に変換する |
| **定量化** | 実コードの計測データ（ファイルサイズ、行数、実行時間）を根拠に組み込む |
| **代替案の提示** | 採用案だけでなく不採用案とその理由を明示する。「なぜやらないか」は「なぜやるか」と同じくらい重要 |
| **過剰設計の抑止** | ROI を意識し、「数十 ms の最適化のために数百行の改修」のような過剰投資を指摘する |
| **scope の判定** | 判断がどの領域（arch / ui / pol 等）に影響するかを明示する |
| **再検討条件の設定** | 「いつこの判断を見直すべきか」を具体的な条件（数値閾値 / 要件変更 / 規模変化）で記述する |
| **実装と検証** | GDR に基づいてコードを書き、テスト・ビルドで検証し、GDR のステータスを更新する |
| **変更の永続化** | 実装・文書変更を適切な粒度でバージョン管理に反映する（例: git commit）。変更メッセージの作成、対象ファイルの選別、反映単位の判断は AI が行う。人間は反映の承認に集中する |

## 7. 前提定義書（Context Definitions）

GDR ビルドアップを開始する前に、プロジェクトの scope と PREFIX を定義する。`templates/T1_CONTEXT_DEFINITIONS.md` をコピーして作成する。

プロジェクト初回の対話で AI と一緒に作成するのが自然な流れであり、scope と PREFIX の選定自体が最初の GDR（例: `GDR-META-001`）になりうる。

定義書はプロジェクトの成長とともに更新する。新しいドメインが出現したら PREFIX を追加し、新しい影響領域が識別されたら scope を追加する。

## 8. ビルドアップ記録

サイクルの対話履歴と判断の積み上がりを記録する文書。前提定義書とは独立に、サイクルを回すたびに追記していく。`templates/T2_BUILDUP_RECORD.md` をコピーして作成する。

セッション単位・スプリント単位・マイルストーン単位など、チームに合った粒度で記録する。記録の目的は「どのような対話からどのような判断が生まれたか」の追跡可能性であり、議事録ではない。

## 9. 導入方法

1. プロジェクトに `documents/GDR/` を配置する
2. 前提定義書（Context Definitions）を `templates/T1_CONTEXT_DEFINITIONS.md` から作成する
3. AI との最初の対話で scope と PREFIX を確定する（これ自体が最初の GDR になる）
4. 以降は「要件を伝える → AI が GDR を構造化 → レビュー → 実装」のサイクルを回す
5. ビルドアップ記録にサイクルの流れを追記する
6. `GDR_INDEX` にレコードが積み上がっていく
7. 判断に迷った時は過去の GDR を参照する。再検討条件に該当するなら更新する
