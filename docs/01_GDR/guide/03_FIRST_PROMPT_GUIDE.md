## 1. ローカライズとは

GDR フレームワーク自体は scope や PREFIX の具体的な定義に依存しない汎用設計である。**ローカライズ**とは、この汎用フレームワークを特定のプロジェクトに合わせてカスタマイズする作業を指す。

具体的には以下を決定する:

- プロジェクトで使う **scope**（影響領域の分類）
- プロジェクトで使う **PREFIX**（ドメイン識別子）
- GDR の **出力先**（ディレクトリ構成）
- **短縮キーワード** などの運用設定

これらの決定自体が最初の GDR（例: `GDR-META-001`）になる。

## 2. 推奨プロンプト

```
GDRを導入する。以下のプロジェクト情報をもとに前提定義書を作成して。

プロジェクト名: {プロジェクト名}
ドメイン: {ソフトウェア / 製造 / 医療 / 金融 / その他}
概要: {プロジェクトの概要を1〜3文で}
主な関心事: {例: API設計、画面UI、運用ルール、コスト管理 など}
GDR出力先: {例: documents/decisions/ documents/kaizen/}

短縮キーワード:
  "改善提案[：|；][タイトル]": T2テンプレートの構成に従って改善提案文書を生成
  "レビュー[：|；][ファイル名]": 同位置にファイル名_reviewedとしてレビュー文書を生成
  "レビュー反映[：|；][ファイル名]": _reviewedの内容を元ファイルに組み込み、_reviewedを削除
  "バグレポート[：|；][ファイル名]": {出力先}/bug_fix_report/ 配下にバグレポートを作成
  {プロジェクト固有のキーワードを追加}
```

このプロンプトにより AI は:

- [T1_CONTEXT_DEFINITIONS.md](/01_GDR/templates/T1_CONTEXT_DEFINITIONS.md) テンプレートに基づいた前提定義書を生成する
- [scope ガイド](/01_GDR/guide/01_SCOPE_GUIDE.md) / [PREFIX ガイド](/01_GDR/guide/02_PREFIX_GUIDE.md) を参照してプロジェクトに適した scope・PREFIX を提案する
- 出力先ディレクトリの構成を提案する
- CLAUDE.md の `コンテキスト短縮用キーワード` セクションを生成する

### 2.1. GDR 出力先の書式

`mkdir` コマンドと同じ感覚でディレクトリを指定する。スペース区切りで複数指定でき、ブレース展開も使える。

```
# スペース区切りで複数指定
GDR出力先: documents/decisions/ documents/kaizen/ documents/bug_fix_report/

# ブレース展開で共通プレフィックスをまとめる
GDR出力先: documents/{decisions,kaizen,bug_fix_report}/
```

AI は指定されたディレクトリごとに用途を推測し、前提定義書の出力先テーブルとして整理する。

## 3. プロンプトの構成要素

| 要素 | 必須 | 目的 | 記述のコツ |
|---|---|---|---|
| プロジェクト名 | ○ | GDR の識別 | 正式名称でなくて良い |
| ドメイン | ○ | 業種別ガイドの参照 | 複数にまたがる場合はカンマ区切り |
| 概要 | ○ | scope・PREFIX 提案の文脈 | 1〜3 文で「何をするプロジェクトか」 |
| 主な関心事 | − | scope の優先順位付け | 意思決定が多い領域を列挙 |
| GDR 出力先 | − | ディレクトリ構成の決定 | スペース区切りまたはブレース展開で複数指定可。省略時は AI が提案 |
| 短縮キーワード | − | CLAUDE.md の短縮キーワード生成 | 既定 4 種（改善提案・レビュー・レビュー反映・バグレポート）＋プロジェクト固有を追加 |
| 既存の制約 | − | 提案の精度向上 | モノレポ構成、既存の ADR など |

## 5. アンチパターン

| パターン | 問題 | 改善 |
|---|---|---|
| scope と PREFIX を自分で全部定義してから渡す | AI の構造化能力を活かせない。過剰定義になりやすい | 概要と関心事だけ渡し、AI に提案させる |
| プロジェクトの詳細を長文で書く | AI が情報過多で焦点を見失う | 1〜3 文の概要 + 関心事のキーワード列挙に絞る |
| ドメインを書かない | 業種別ガイドを参照できず、汎用的すぎる提案になる | 一言でよいのでドメインを明記する |
| 最初から完璧を目指す | ビルドアップの思想に反する。手が止まる | 粗くてよいので対話を始める。磨くのは後 |
