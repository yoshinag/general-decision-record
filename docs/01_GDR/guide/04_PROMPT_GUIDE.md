## 1. 要件伝達

サイクルの起点。課題や要件を AI に伝え、GDR の構造化を開始する。

| プロンプト | 効果 |
|---|---|
| `[課題/要件の説明]` | 自然言語で課題を伝える。AI が GDR として構造化する |
| `改善提案：[タイトル]` | GDR → 実行計画 → ふりかえりの 3 部構成で改善提案文書を生成する |
| `[課題]。GDR化して` | 口頭レベルの課題を明示的に GDR フォーマットへ変換させる |
| `バグレポート：[内容]` | バグ内容を整理したレポート文書を生成する |

## 2. 構造化・深掘り

AI の初期提案に対して、判断の精度を上げるための深掘り。

| プロンプト | 効果 |
|---|---|
| `代替案を出して` | 不採用案とその理由を含めた比較表を求める |
| `scope は？` | 判断の影響領域を明示させる |
| `再検討条件を具体的に` | 抽象的な条件を数値閾値や具体的トリガに落とし込ませる |
| `定量化して` | 実測データや数値根拠を求める |
| `影響範囲は？` | 変更が及ぶファイル・モジュール・チームを列挙させる |
| `[案A] と [案B] を比較して` | 特定の代替案同士のトレードオフを整理させる |

## 3. レビュー

提案や文書の品質を検証し、過剰設計を刈り込む。

| プロンプト | 効果 |
|---|---|
| `レビュー：[ファイル名]` | 対象ファイルのレビュー文書を生成する |
| `レビュー反映：[ファイル名]` | レビュー内容を元ファイルに組み込み、レビュー文書を削除する |
| `過剰じゃない？` | ROI 観点で提案を圧縮させる |
| `最適解を` | 過剰設計を刈り込むトリガ。最小限の手段で目的を達成する案を求める |
| `ガイドに沿って編集して` | GDR 書式の必須フィールド漏れやフォーマット逸脱を修正させる |

## 4. 実装

GDR を具体的なコード・設定・文書に落とし込む。

| プロンプト | 効果 |
|---|---|
| `implements it` | 直前の提案・GDR を実装に反映させる |
| `タスク化して` | GDR を実行可能なタスク（フェーズ・チェックリスト）に分解させる |
| `採番しなおして` | ファイル名・見出し・相互リンクの採番を整合させる |
| `コミット` | 変更をコミットさせる |

## 5. ふりかえり・検証

サイクル終了時の確認とナレッジの定着。

| プロンプト | 効果 |
|---|---|
| `ふりかえり` | T2 のふりかえり章（定量サマリー・傾向・申し送り）を生成する |
| `整合性チェック` | ファイル間のリンク・採番・パスの整合性を検証させる |
| `文書の整合性を整理して` | 不整合を検出し修正まで実行させる |
| `[ファイル名]を再編集` | 現在のテンプレート構成や運用ルールに合わせて文書を書き直させる |

## 6. コンテキスト短縮キーワード（CLAUDE.md 連携）

CLAUDE.md（プロジェクトルートの AI エージェント向け指示ファイル）に**短縮キーワード**を定義しておくと、一言で定型操作をトリガできる。キーワードは AI がセッション開始時に読み込むため、毎回プロンプトで説明する必要がない。

### 定義済みキーワード

| キーワード | 書式 | 動作 |
|---|---|---|
| **改善提案** | `改善提案：[タイトル]` | T2_BUILDUP_RECORD.md テンプレートの構成に従って改善提案文書を生成する |
| **レビュー** | `レビュー：[ファイル名]` | 対象ファイルと同じ位置に `{ファイル名}_reviewed` としてレビュー文書を生成する |
| **レビュー反映** | `レビュー反映：[ファイル名]` | `_reviewed` ファイルの内容を検討・整理して元ファイルに組み込み、`_reviewed` ファイルを削除する |
| **バグレポート** | `バグレポート：[ファイル名]` | `documents/bug_fix_report/` 配下にバグ内容を整理したレポート文書を作成する |

### 仕組み

- `コンテキスト短縮用キーワード` セクションにキーワードと動作を定義する
- AI は**セッション開始時に CLAUDE.md を読み込む**ため、キーワードだけで意図が伝わる
- プロジェクト固有のキーワードを自由に追加できる

### カスタマイズ例

プロジェクトに合わせてキーワードを追加する:

```markdown
## コンテキスト短縮用キーワード

5. **"リリースノート[：|；][バージョン]":** CHANGELOG.md に対象バージョンの変更履歴をまとめる。
6. **"設計相談[：|；][テーマ]":** 代替案・トレードオフを含む設計検討メモを生成する。
```

## 7. 使い方のコツ

1. **短く言う** — 効果的なプロンプトは短い。`implements it`、`コミット`、`最適解を` のように一言で十分
2. **組み合わせる** — `改善提案：X` → `過剰じゃない？` → `implements it` → `コミット` のようにサイクルを回す
3. **迷ったら聞く** — 判断材料が足りない時は `代替案を出して` や `定量化して` で情報を引き出してから決める
4. **事後的でよい** — GDR は対話の中で結晶化するもの。最初から完璧なプロンプトを書く必要はない
5. **文書と実装を混ぜない** — 1 回の指示で「文書作成」と「コード実装」を同時に頼まない。文書を先に作成・レビューし、合意を得てから実装に進む。混ぜると判断が曖昧なまま実装が走り、手戻りが大きくなる
