## 1. GDR（General Decision Record）

**GDR-META-012: .gitignore をプロジェクト実態に合わせて刷新する**

- **status:** Implemented
- **scope:** meta, pol
- **決定:** 現在の .gitignore は Textpattern CMS 由来の不要なエントリで構成されている。これを削除し、本プロジェクトに必要なエントリ（`.idea/`, `.DS_Store` 等）に置き換える
- **理由:** 不要なエントリが残っていると、プロジェクトの技術スタックについて誤解を招く。また `.idea/` が untracked のまま放置されており、誤コミットのリスクがある
- **影響:** .gitignore の内容を全面的に書き換える
- **再検討条件:** 技術スタックが変わった場合

**GDR-META-013: markdownlint による文書品質の自動チェックを導入する**

- **status:** Implemented
- **scope:** meta, pol
- **決定:** markdownlint-cli2 をシステムにインストールし、`.markdownlint.json` でプロジェクト固有のルールを設定する。リポジトリにコミットするのは設定ファイルのみ。インストール方法はプラットフォームに応じて選択する（macOS: `brew`、Linux: `snap` / バイナリ、クロスプラットフォーム: `npx` による一時実行）
- **理由:** 文書量が増えるにつれ、見出しレベルの不整合、空行不足、インデントのばらつきが手動では検出しにくくなる。GDR 文書は構造が重要であり、フォーマットの一貫性が AI の解析精度に直結する。設定ファイルのみをコミットすることで、リポジトリ自体はランタイム非依存を維持する
- **影響:** `.markdownlint.json` をリポジトリルートに追加する。既存文書のうちルール違反があるものは修正する
- **再検討条件:** ルールが厳しすぎて文書作成の妨げになった場合は緩和する

**GDR-META-014: .editorconfig でファイル書式を統一する**

- **status:** Implemented
- **scope:** meta, pol
- **決定:** `.editorconfig` を導入し、文字コード（UTF-8）、改行コード（LF）、インデント（スペース 2）、末尾改行の有無をリポジトリ単位で統一する
- **理由:** AI（Claude Code 等）、エディタ（VS Code, JetBrains）、GitHub 上の編集など、文書を編集する主体が複数存在する。主体ごとにデフォルト設定が異なると、改行コードの混在やインデント不一致が発生し、diff が無意味に膨らむ。`.editorconfig` は対応エディタ・ツールが自動的に読み取るため、設定の共有コストがゼロ
- **影響:** `.editorconfig` をリポジトリルートに追加する
- **再検討条件:** `.editorconfig` 非対応のツールが主要な編集手段になった場合

**GDR-META-015: シェルスクリプトで文書間リンクの整合性を検証する**

- **status:** Implemented
- **scope:** meta, pol
- **決定:** markdown 内のローカルリンク（`[text](path)`）を抽出し、リンク先ファイルの存在を検証するシェルスクリプト `scripts/check-links.sh` を作成する。POSIX 互換のコマンドのみ使用し、macOS / Linux の両環境で動作させる
- **理由:** このセッション中にディレクトリ移動（`01_GDR/markdown/` → `01_GDR/`）が発生し、`_sidebar.md`・README・CLAUDE.md のパス更新が必要になった。AI がリネーム・移動を行った後に `bash scripts/check-links.sh` を実行すればリンク切れを自力で検出できる。シェルスクリプトなら追加の依存なしでどの環境でも動作する
- **影響:** `scripts/check-links.sh` を追加する
- **再検討条件:** リンク構造が複雑化し、シェルスクリプトでは対応しきれなくなった場合（アンカーリンクの検証等）

## 2. 現状

1. `.gitignore` が Textpattern CMS 由来のエントリのままで、`.idea/` 等の IDE 設定が除外されていない
2. markdown の品質チェック（lint）がなく、フォーマットの不整合が手動レビューでしか検出できない
3. `.editorconfig` がなく、編集主体ごとに改行コード・インデントが異なるリスクがある
4. 文書間のリンク整合性を検証する手段がなく、ディレクトリ移動やリネーム時にリンク切れが発生しやすい


---

## 3. 改善案

プロジェクトの文書管理品質を向上させるため、4 つのツール/設定を段階的に導入する。リポジトリにコミットするのは設定ファイルとシェルスクリプトのみで、特定のランタイムに依存しない:

1. **.gitignore の刷新 + .editorconfig** — 設定ファイルの追加のみ。即時対応可能
2. **markdownlint** — プラットフォーム別にインストール（brew / snap / npx）。設定ファイルのみコミット
3. **check-links.sh** — POSIX 互換シェルスクリプトによるリンク検証。外部依存なし

### 4.2. .editorconfig

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true

[*.md]
indent_style = space
indent_size = 2
trim_trailing_whitespace = false
```

- `trim_trailing_whitespace = false`（markdown）: markdown の末尾スペース 2 つによる改行（`<br>`）を保持するため

### 4.3. markdownlint

インストール（プラットフォーム別）:

| 環境 | コマンド |
|---|---|
| macOS | `brew install markdownlint-cli2` |
| Linux | `snap install markdownlint-cli2` または[リリースバイナリ](https://github.com/DavidAnson/markdownlint-cli2/releases) |
| どこでも（Node.js あり） | `npx markdownlint-cli2 'documents/**/*.md'`（一時実行、インストール不要） |

`.markdownlint.json`（リポジトリにコミット）:

```json
{
  "MD013": false,
  "MD033": false,
  "MD041": false
}
```

- `MD013`（行長制限）: 日本語文書では無効化
- `MD033`（HTML 禁止）: Docsify で HTML を使う可能性があるため無効化
- `MD041`（最初の行が h1）: テンプレートの `>` 注釈行を許容するため無効化

実行:

```bash
markdownlint-cli2 'documents/**/*.md'
```

### 4.4. check-links.sh（リンク整合性チェック）

`scripts/check-links.sh`（POSIX 互換: macOS / Linux 両対応）:

```bash
#!/bin/bash
# markdown 内のローカルリンクを検証する（GNU/BSD grep 両対応）
set -euo pipefail

errors=0

check_file() {
  local file="$1"
  local dir
  dir=$(dirname "$file")

  # [text](path) からパス部分を抽出（# アンカーと外部 URL を除外）
  sed -n 's/.*\[.*\](\([^)#]*\)).*/\1/p' "$file" | while read -r link; do
    # 空文字・外部 URL はスキップ
    [ -z "$link" ] && continue
    case "$link" in http://*|https://*) continue ;; esac

    local target="$dir/$link"
    if [ ! -f "$target" ]; then
      echo "BROKEN: $file -> $link"
    fi
  done
}

# 対象ファイルを検証
find documents -name '*.md' -print0 | while IFS= read -r -d '' file; do
  check_file "$file"
done
for file in README.md CLAUDE.md; do
  [ -f "$file" ] && check_file "$file"
done
```

AI がディレクトリ移動やリネームを行った後に実行:

```bash
bash scripts/check-links.sh
```

## 5. 検討課題

1. **markdownlint の既存文書への影響**
   - 解決策: まず `--fix` オプションで自動修正可能な違反を一括修正し、残りを手動対応する
   - → フェーズ 2 のタスク 2.2 として実行計画に追加済み
2. **check-links.sh の検証範囲**
   - 解決策: ファイル存在チェックのみとし、アンカーリンク（`#section`）の検証は対象外とする。アンカー検証が必要になった場合はツール導入を再検討する
   - → GDR-META-015 の再検討条件として記録済み
3. **markdownlint のマルチプラットフォーム対応**
   - 解決策: リポジトリには設定ファイル（`.markdownlint.json`）のみをコミットし、ツール自体のインストールは各環境に委ねる。macOS: brew、Linux: snap / バイナリ、一時実行: npx。いずれの方法でも同じ設定ファイルを参照する
   - → 詳細セクション 4.3 にプラットフォーム別のインストール手順を記載済み
