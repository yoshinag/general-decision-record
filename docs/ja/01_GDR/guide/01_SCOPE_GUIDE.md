# 01. scope ガイドライン

GDR の `scope` は決定の影響領域を示す分類である。scope の選定はプロジェクトや業界によって異なるが、以下に共通的なものと業種別の例を示す。

## 1. 共通 scope

業種を問わず多くのプロジェクトで利用される scope。

| scope | 略記 | 対象 | 判断の例 |
|---|---|---|---|
| `architecture` | `arch` | システム構成、データフロー、レイヤー境界、DB スキーマ | コンポーネント分割、通信方式の選定、データ永続化方式 |
| `policy` | `pol` | 開発プロセス、コーディング規約、リリース方針 | エラー処理方針、ブランチ戦略、レビュー基準 |
| `product` | `prod` | プロダクト方針、命名、対象ユーザー、収益モデル | アプリ名称、対象年齢、課金モデル |
| `performance` | `perf` | パフォーマンス目標、最適化方針、計測戦略 | 応答時間目標、起動速度基準、計測ポイント |
| `meta` | `meta` | GDR 運用ルール自体の変更、ドキュメント構造 | scope の追加、PREFIX の改廃、書式変更 |

## 2. ソフトウェア開発

モバイルアプリ・Web アプリ・バックエンドサービス等に特有の scope。

| scope | 略記 | 対象 | 判断の例 |
|---|---|---|---|
| `spec` | `spec` | 機能仕様、データモデル、API 契約 | エンドポイント設計、データフォーマット、画面遷移仕様 |
| `ui` | `ui` | ビジュアルデザイン、レイアウト、インタラクション | フォントサイズ基準、レスポンシブ方針、アニメーション有無 |
| `testing` | `test` | テスト戦略、カバレッジ目標、E2E 方針 | ユニットテスト範囲、モック方針、CI テスト構成 |
| `security` | `sec` | 認証・認可、暗号化、脆弱性対応 | 認証方式の選定、データ暗号化方針、依存ライブラリ監査 |
| `data` | `data` | データ管理、バックアップ、保持期間 | マイグレーション戦略、バックアップ頻度、個人情報保持方針 |

## 13. 導入ガイド

プロジェクトに scope を導入する際:
1. 本ガイドの共通 scope + 該当業種セクションから scope を選定する
2. 複数業種にまたがるプロジェクトでは各業種から必要な scope を組み合わせる
3. プロジェクト固有の scope が必要であれば追加定義する
4. 使用する scope を Context Definitions（前提定義書）に記載する
5. 1 つの GDR が複数の scope を持つことがある（例: `arch, pol`）
