## 1. プロジェクトへの導入

GDR フレームワークを新規または既存プロジェクトに導入するためのプロンプト。submodule を参照元として AI に文書基盤を構築させる。

### 前提条件

```bash
git submodule add <url> submodules/general-decision-record
```

### 基盤構築（新規・既存共通）

| プロンプト | 効果 |
|---|---|
| `submodules/general-decision-record を参照して、このプロジェクトに GDR フレームワークを導入して` | T1 前提定義書の作成、CLAUDE.md への GDR 書式・採番ルール・キーワード追加を一括で実行する |

AI は以下を生成する:

1. **T1 前提定義書** — プロジェクトのドメインに合った scope を選定し、既存の設計判断領域から PREFIX を抽出する。scope と PREFIX の選定自体が最初の GDR（`GDR-META-001`）になる
2. **CLAUDE.md の GDR 運用セクション** — GDR 書式（6 フィールド）、採番ルール、ビルドアップサイクルの概要をインライン記載し、詳細は submodule への参照リンクで補完する
3. **キーワード体系** — 本ガイドのプロンプトを CLAUDE.md に追加する

### 既存文書の整合（既存プロジェクトの場合）

| プロンプト | 効果 |
|---|---|
| `既存の改善提案/設計文書を GDR フレームワークに沿って再構成して` | テンプレートの統一、重複文書の廃止を改善提案として起案する |

既存プロジェクトでは独自テンプレートや文書構成が存在することがある。その場合:

- 改善提案テンプレートを T2 簡素化版（GDR → 実行計画 → ふりかえり）に統一する
- submodule と重複する定義ファイルは廃止し、submodule を正とする
- 既存文書は遡及的に書き換えない。新規分から適用する

### 設計のポイント

| # | ポイント | 理由 |
|---|---|---|
| 1 | submodule は「参照」であり「編集対象」ではない | フレームワークの正本は submodule に閉じ、プロジェクト固有の判断はプロジェクト側の文書に記録する |
| 2 | CLAUDE.md にインライン記載する情報は「AI が即座に使える最低限」に留める | 詳細は submodule 参照で済ませる 2 層構造。CLAUDE.md の肥大化を防ぐ |
| 3 | 前提定義書（T1）の作成自体が最初の GDR になる | scope と PREFIX の選定は設計判断であり、GDR-META-001 として記録する |
| 4 | 既存文書は遡及的に書き換えない | コストに見合わない。新規分から新テンプレートを適用すれば自然に統一される |

## 2. 要件伝達

サイクルの起点。課題や要件を AI に伝え、GDR の構造化を開始する。

| プロンプト | 効果 |
|---|---|
| `[課題/要件の説明]` | 自然言語で課題を伝える。AI が GDR として構造化する |
| `改善提案：[タイトル]` | GDR → 実行計画 → ふりかえりの 3 部構成で改善提案文書を生成する |
| `[課題]。GDR化して` | 口頭レベルの課題を明示的に GDR フォーマットへ変換させる |
| `バグレポート：[内容]` | バグ内容を整理したレポート文書を生成する |

## 3. 構造化・深掘り

AI の初期提案に対して、判断の精度を上げるための深掘り。

| プロンプト | 効果 |
|---|---|
| `代替案を出して` | 不採用案とその理由を含めた比較表を求める |
| `scope は？` | 判断の影響領域を明示させる |
| `再検討条件を具体的に` | 抽象的な条件を数値閾値や具体的トリガに落とし込ませる |
| `定量化して` | 実測データや数値根拠を求める |
| `影響範囲は？` | 変更が及ぶファイル・モジュール・チームを列挙させる |
| `[案A] と [案B] を比較して` | 特定の代替案同士のトレードオフを整理させる |

## 4. レビュー

提案や文書の品質を検証し、過剰設計を刈り込む。

| プロンプト | 効果 |
|---|---|
| `レビュー：[ファイル名]` | 対象ファイルのレビュー文書を生成する |
| `レビュー反映：[ファイル名]` | レビュー内容を元ファイルに組み込み、レビュー文書を削除する |
| `過剰じゃない？` | ROI 観点で提案を圧縮させる |
| `最適解を` | 過剰設計を刈り込むトリガ。最小限の手段で目的を達成する案を求める |
| `ガイドに沿って編集して` | GDR 書式の必須フィールド漏れやフォーマット逸脱を修正させる |

## 5. 実装

GDR を具体的なコード・設定・文書に落とし込む。

| プロンプト | 効果 |
|---|---|
| `implements it` | 直前の提案・GDR を実装に反映させる |
| `タスク化して` | GDR を実行可能なタスク（フェーズ・チェックリスト）に分解させる |
| `採番しなおして` | ファイル名・見出し・相互リンクの採番を整合させる |
| `コミット` | 変更をコミットさせる |

## 6. ふりかえり・検証

サイクル終了時の確認とナレッジの定着。

| プロンプト | 効果 |
|---|---|
| `ふりかえり` | T2 のふりかえり章（定量サマリー・傾向・申し送り）を生成する |
| `整合性チェック` | ファイル間のリンク・採番・パスの整合性を検証させる |
| `文書の整合性を整理して` | 不整合を検出し修正まで実行させる |
| `[ファイル名]を再編集` | 現在のテンプレート構成や運用ルールに合わせて文書を書き直させる |

---

## 7. 使い方のコツ

1. **短く言う** — 効果的なプロンプトは短い。`implements it`、`コミット`、`最適解を` のように一言で十分
2. **組み合わせる** — `改善提案：X` → `過剰じゃない？` → `implements it` → `コミット` のようにサイクルを回す
3. **迷ったら聞く** — 判断材料が足りない時は `代替案を出して` や `定量化して` で情報を引き出してから決める
4. **事後的でよい** — GDR は対話の中で結晶化するもの。最初から完璧なプロンプトを書く必要はない
